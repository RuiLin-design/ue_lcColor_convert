Deep Dive into UE5 Color Space:
The Love-Hate Relationship Between Linear Color and sRGB

UI/UX Technical Artist
Published Apr 5, 2026

In Unreal Engine 5 (UE5) UI development and material creation, inaccurate colors are a perpetual issue. When UI designers hand off hexadecimal (HEX) colors from Figma, simply plugging them into UE5 often results in colors that look "washed out" or "over-brightened." The ultimate culprit behind this is the discrepancy in Color Space.

Why Does the Color Shift Occur?

To understand this, we need to look at how monitors and game engines fundamentally interpret "light":

  • sRGB (Gamma 2.2): To align with human visual perception (we are far more sensitive to details in shadows), our monitors default to using a non-linear sRGB curve to display colors. Colors in Figma and Photoshop are natively based on this space.
  • Linear Color (Linear Space): Modern game engines like UE5 rely heavily on Physically Based Rendering (PBR). To ensure that lighting and color blending are mathematically and physically precise, the engine forces all internal calculations to be processed in a "Linear Space."

If you feed an sRGB value like RGB(128, 128, 128) directly into a UE5 material node without conversion, the engine treats it as pure linear physical light intensity. When it eventually outputs to your screen, it gets hit with the monitor's Gamma curve again, causing the colors to appear pale and desaturated.

A Historical Artifact: Why Was Gamma Invented?

Many wonder: if linear space calculation is more accurate, why didn't we just use it from the beginning? This is actually a "technical debt" left over from legacy CRT monitors. Early cathode-ray tube displays had a natural, non-linear relationship (roughly a 2.2 power curve) between input voltage and output brightness.

Coincidentally, human perception of brightness is also non-linear (Weber-Fechner law). We are extremely sensitive to changes in dark areas, but somewhat oblivious to variations in highlights. Consequently, engineers embraced this "flaw" and used this non-linear characteristic (Gamma encoding) to store images, dedicating precious data bandwidth to shadow details. This standard has persisted to this day, becoming the sRGB standard we both love and hate.

The Core: Reverse Gamma Correction

To make colors in the game look exactly like your design mockups, we must perform a Reverse Gamma Correction (sRGB to Linear). In the underlying code, this isn't just multiplying by a simple constant; it involves a standard mathematical piecewise function. Here is the core conversion algorithm in JavaScript: 

// Converts 0-1 range sRGB channel values to Linear values
function sRGBToLinear(s) {
    if (s <= 0.04045) {
        return s / 12.92;
    } else {
        return Math.pow((s + 0.055) / 1.055, 2.4);
    }
}

Practical Guidelines in Unreal Engine

When you receive UI slices or color palettes, never punch in numbers blindly. Depending on your development context, the approach varies:

1. Material Editor

If you've imported a UI texture, you must verify the sRGB checkbox in the texture details panel.

  • Base Color Maps: Visual textures like UI icons and color illustrations must have sRGB checked. The engine will automatically convert them to Linear upon sampling.
  • Data Maps: Textures used to store physical data like Mask, Roughness, Metallic, or Normal maps must have sRGB unchecked. Otherwise, the engine incorrectly applies Gamma correction, completely breaking the material's physical properties.
💡 TA Tip: If you are using a VectorParameter (color node) directly in a material, notice the small sRGB checkbox in the upper right corner of the color picker. Check it if you are pasting a Figma HEX code; uncheck it if you are inputting pure physical light intensity.

2. C++ and Blueprints

At the code level, you can use the Make Color node in Blueprints (which handles the conversion under the hood). In C++, the standard practice is to parse the sRGB HEX string first, and then convert it into an FLinearColor struct for rendering:

// Create accurate linear color from designer's HEX string
FColor sRGBColor = FColor::FromHex(TEXT("3EAE70"));
FLinearColor LinearColor = FLinearColor(sRGBColor);

The Final Boss: ACES Tonemapping and UI Color Shift

Even if you perfectly master the sRGB to Linear conversion, you might still notice something: When the UI is rendered in the 3D world, the colors are still wrong!

This introduces UE5's other final boss — ACES (Academy Color Encoding System) Cinematic Tonemapping. To give the scene a cinematic feel, UE5's Post Process Volume applies a final "color grade" to all rendered light. This color grading causes extremely vivid colors (especially high-saturation reds and blues) to shift.

How to fix this?

  1. For 2D UI (UMG): UMG is rendered by default overlaid on top of the screen after the scene is finished. It is unaffected by scene post-processing, so as long as your colors are correct in UMG, they will display correctly.
  2. For 3D UI (e.g., VR interfaces, world-space widgets): If your UI is placed in the scene as a 3D object, it will be heavily impacted by ACES. The solution is to change the UI material's Material Domain to User Interface. This forces the material to bypass complex scene lighting and tonemapping, outputting the raw, original color.

Hopefully, this article helps you completely demystify colors in Unreal Engine. Only by keeping the underlying data pure and physically precise can we render the most perfect visuals!
